# Guardrail Fabric v0.2 Breaking-Change Policy

All schemas in this repository are at `v0.1`. This document defines the breaking-change
policy for the first version bump to `v0.2`.

---

## Scope

This policy covers:

| Schema | Version | Consumed by |
|--------|---------|-------------|
| `trustops-guardrail-action-decision.v0.1.schema.json` | v0.1 | `model-router` (guardrail decision refs), `model-governance-ledger` (evidence records), `prophet-platform` validators, `prophet-cli` `prophet guardrail test` facade |
| `sourceos.guardrail.decision.v0.1.schema.json` | v0.1 | SourceOS carry pipeline, `agent-registry` authority checks |
| `professional-intelligence-guardrail-pack.schema.json` | v0.1 | `prophet-platform` Professional Intelligence validators |

---

## What constitutes a breaking change

### `trustops-guardrail-action-decision.v0.1`

A breaking change is any of:

- Adding a new required field (`schemaVersion`, `recordType`, `decision_id`, `source_system`,
  `controlling_outcome`, `runtime_action`, `receipt_ids`, `gate_ids`, `evidence_refs`,
  `policy_refs`, `reason`, `issued_at`, `authority_mutation`, `agentplane_projection` are
  required today â€” any addition to this list breaks existing producers)
- Changing the type, format, or allowed values of any required field
- Changing the `controlling_outcome` enum â€” any removal of `allow`, `deny`, or
  `requires-review` breaks downstream routing logic
- Changing the `authority_mutation` or `agentplane_projection` structure â€” these are
  consumed by the agentplane adapter contract and the prophet-mesh runtime release bundle
- Renaming the schema file path â€” all `policy_refs` and evidence records reference the
  path literally

### `sourceos.guardrail.decision.v0.1`

A breaking change is any of:

- Removing or renaming fields referenced by SourceOS carry pipeline selection logic
- Changing the `controlling_outcome` value space used in carry admission filtering

### `professional-intelligence-guardrail-pack.schema.json`

A breaking change is any of:

- Changing required fields consumed by `prophet-platform` validator
  (`validate_professional_intelligence.py` or equivalent)
- Changing pack-level metadata fields referenced in the bundle registry

---

## What is NOT a breaking change

- Adding optional fields (not in the `required` array)
- Adding new enum values to `controlling_outcome` that downstream consumers are required
  to treat as deny-safe defaults
- Adding new optional evidence or label fields
- Updating `description` strings without changing field semantics

---

## Version-bump procedure

1. Create a new schema file at `v0.2` (e.g. `trustops-guardrail-action-decision.v0.2.schema.json`)
2. Update `schemaVersion` field's `const` or `enum` in the new file
3. Open coordinated PRs in all consuming repos before merging the v0.2 schema:
   - `model-router`: update `model-catalog-entry.v0.1.ts` `guardrailPolicyRef` docs and
     any validator that pins schema path
   - `model-governance-ledger`: update evidence record examples
   - `prophet-platform`: update validator script imports / path references
   - `prophet-mesh`: update `specs/` YAML refs
4. Do not delete or modify the v0.1 schema file until all consumers confirm migration
5. A `content_sha256` mismatch between two repos referencing the same schema path must be
   flagged explicitly â€” never silently resolved

---

## Current state

No schema has reached the v0.2 gate. Proceed with v0.2 only after:

- At least one production-path consumer (model-router or prophet-platform) has filed an
  explicit schema change request with a concrete field-level diff
- Coordinated PR readiness confirmed in all consuming repos listed above
